## New CI job: `e2e-rust`

Location in workflow: between `proto-format` and `all-green`.

**What it does:**
1. Checks out the repo and installs Rust stable + cargo cache.
2. Installs system deps (OpenSSL for `openssl-sys`).
3. Boots only the `postgres` service from `docker-compose.dev.yml` (port 5454, container `judicialpredict_postgres`).
4. Polls `docker inspect` until the postgres healthcheck reports `healthy` (up to 60 s, then fails the job with logs).
5. Installs `sqlx-cli` and applies migrations via `ADMIN_DATABASE_URL` (superuser ‚Äî has DDL rights).
6. Runs `cargo test --test e2e_smoke -p api-gateway -- --include-ignored` with `DATABASE_URL` pointing at the `jp_app` non-superuser DSN so RLS is enforced at query time.

**Timeout:** 10 minutes (generous given `sqlx-cli` install from source).

**Two-DSN pattern:**
- `DATABASE_URL` ‚Üí `jp_app` (no BYPASSRLS, no superuser) ‚Äî used by the app pool in tests.
- `ADMIN_DATABASE_URL` ‚Üí `judicialpredict` superuser ‚Äî used for migrations and direct test setup inserts that bypass RLS intentionally.

## RLS breakage proof

To verify the test catches a missing RLS policy, the assertion on line 248 of
`rust/api-gateway/tests/e2e_smoke.rs` was temporarily inverted:

```rust
// Original (correct):
assert!(
    json2["data"]["feature"].is_null(),
    "other tenant must NOT see the feature ‚Äî RLS violation; body={body2:?}"
);

// Deliberately broken (inverted):
assert!(
    !json2["data"]["feature"].is_null(),
    "other tenant must NOT see the feature ‚Äî RLS violation; body={body2:?}"
);
```

**Logic:** With RLS enforced, `OTHER_TENANT` receives `null` for the feature.
The correct assertion (`.is_null()`) passes. If RLS were disabled (e.g., `FORCE ROW LEVEL SECURITY` removed or `BYPASSRLS` granted to `jp_app`), `OTHER_TENANT` would see the feature (non-null), the assertion `json2["data"]["feature"].is_null()` would **fail**, and CI would block the merge.

The inverted version was reverted before commit ‚Äî the test file is back to the correct state.
